# Правила для агентов (msk-go)

Вы работаете в песочнице — виден ТОЛЬКО этот репозиторий. К другим проектам и
системе доступа нет (by design). Цель проекта: максимум автоматизации, минимум
участия человека.

## Поток работы (AI-компания)
1. **CEO** декомпозирует задачу и раздаёт под-задачи команде (skill `paperclip-board`).
2. **Engineer (Staff)** делает изменение в **новой ветке** `agent/<краткое-имя>`, коммитит, пушит, открывает PR в `main`.
3. **QA** делает код-ревью PR (skill `review`): корректность, не сломано ли, по делу ли.
4. **Release** мёржит PR **только если**: (а) CI зелёный (workflow `ci`), И (б) QA одобрил. Иначе — вернуть инженеру.
5. Каждый агент закрывает свою задачу статусом **done** по завершении (skill `paperclip-board`), не оставляет `in_progress`.

## Жёсткие правила
- В `main` напрямую НЕ коммитить и НЕ пушить — только через ветку + PR.
- Мёрж — только при зелёном CI. Красный CI = не мёржить, чинить.
- Не выходить за пределы репозитория; не трогать секреты; `git push` только этого репо.
- Без force-push, без переписывания истории, без удаления чужих веток.
- Один PR = одно логическое изменение, осмысленные коммиты.
- Не печатать секреты (токены/ключи из окружения) в комментарии, коммиты, артефакты.

## Как мёржить (Release)
`gh pr merge <num> --squash` после зелёного CI и одобрения QA. Затем закрыть задачу `done`.
